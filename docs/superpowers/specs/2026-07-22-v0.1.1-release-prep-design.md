# v0.1.1 Release Preparation Design

**Date:** 2026-07-22

## Context

PR #9 merged the portable ARM, no-default-login, local-only updater, installer-security, branding, and CI work into `main`. The root README still describes legacy `~/.grok` paths, an unqualified background catalog refresh, xAI sign-in as normal first-run guidance, and incomplete release/privacy behavior. The two release-facing Cargo packages still report `0.1.0`, while the next intended release is `v0.1.1`.

The follow-up should prepare‚Äînot publish‚Äîthe release. It must leave tagging and GitHub release creation for separate explicit approval.

## Scope

Create a focused `release/v0.1.1-prep` branch from current merged `main`.

### README

Update `README.md` to:

- use `~/.groky` for active diagnostics and provider credentials;
- explain `GROKY_HOME` as canonical and `GROK_HOME` / `~/.grok` as compatibility inputs;
- remove the unqualified ‚Äú24-hour background refresh‚Äù statement;
- document that release builds perform no telemetry or automatic update phone-home by default;
- state that provider, explicitly configured auth, and other user-invoked network features still use the network when requested;
- explain that credential-less first run enters the app with a passive `XAI_API_KEY` / `/providers` hint rather than forcing browser login;
- explain that updating means re-running the installer and that `groky update` is inert;
- document Linux prebuilt support for x86_64 and generic ARMv8-compatible aarch64;
- clarify source-build versus prebuilt platform support;
- update roadmap items completed by PR #9;
- avoid claiming that `v0.1.1` artifacts exist before the tag is published;
- use a generic release-tag placeholder rather than presenting `v0.1.0` as the recommended pin.

### Version metadata

Bump both release-facing packages:

- `crates/codegen/xai-grok-pager/Cargo.toml`: `0.1.0` ‚Üí `0.1.1`
- `crates/codegen/xai-grok-pager-bin/Cargo.toml`: `0.1.0` ‚Üí `0.1.1`

Refresh `Cargo.lock` through Cargo. Do not bump `xai-grok-version`; it intentionally retains upstream numbering for internal/client-version compatibility.

## Non-goals

- Do not tag `v0.1.1`.
- Do not create or publish a GitHub release.
- Do not claim that `v0.1.1` is downloadable before release assets exist.
- Do not merge new `upstream/main` monorepo-sync commits.
- Do not broadly rename internal `xai-grok-*`, `grok.com`, `GROK_*`, `/etc/grok`, or migration identifiers.
- Do not expand into a general documentation rewrite.

## Verification

Run serially with low resource pressure and the shared target cache ceiling:

1. `cargo fmt --all -- --check`
2. `bin/check-no-network-telemetry.sh`
3. portable `target-cpu=generic` guard
4. `cargo check -p xai-grok-pager-bin`
5. `cargo test -p xai-grok-update`
6. verify both manifests and lockfile resolve to `0.1.1`
7. if cache remains below the 115 GB pause threshold, run `cargo build --release -p xai-grok-pager-bin`
8. verify `./target/release/groky --version` begins with `groky 0.1.1`
9. run `scripts/test-install.sh`
10. inspect README links and ensure no new default-path guidance uses `~/.grok`

Use `CARGO_BUILD_JOBS=2`, `nice -n 10`, and one Cargo command at a time. Measure `/home/amanverasia/.cache/groky-cargo-target` before and after storage-heavy commands; pause at 115 GB and never exceed 120 GB.

## Delivery

Commit the README and version-prep changes as a focused release-preparation commit, push `release/v0.1.1-prep` to `origin`, and open a PR targeting `main`. Keep the branch/worktree until CI and review are complete. Tagging and publishing remain separate approval gates.
